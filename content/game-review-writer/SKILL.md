# 游戏评测写作

## 输入约定

用户提供**游戏名 + 个人体验要点**（结构化要点或零散笔记均可）：
- 基础信息：开发商、发行年份、平台、自己玩的时长
- 个人感受：喜欢/不喜欢什么系统、整体评价倾向
- 如有版本特定内容（如 S2 复盘），提供版本变化说明

用户没有提供的字段（开发商、年份等），在确认输入时主动询问。

## 工作流程

### 1. 确认输入
确认游戏名称、平台、游玩时长、个人核心感受。如果是**更新已有评测**（同一游戏的版本/赛季复盘），记录更新意图并标记需要读取原文件。

### 2. 学习风格
读取 `content/game-reviews/nzwl.mdx`（深度分析型范式）学习写作风格和 frontmatter 惯例。如果是更新已有评测，额外读取目标 `.mdx` 文件。

### 3. 网络研究
并行执行两次 WebSearch，聚焦：
- **游戏机制**：战斗系统、养成系统、经济系统、关卡设计、数值架构的深度分析
- **玩家社区评价**：Steam/NGA/Reddit/贴吧 等平台的玩家讨论、争议点、版本更新反馈

基础信息（开发商、年份、平台）由用户提供，不必专门搜索。

### 4. 撰写草案
在对话中展示完整评测草案，包含：
- 完整 YAML frontmatter（见 REFERENCE.md 字段参考）
- 按 6 节模板组织的正文（见 REFERENCE.md 章节模板）
- 建议评分（一位小数，附带理由，见 REFERENCE.md 评分标准）

写作要求：
- 用户的个人体验是**权威事实来源**，网络研究只做补充支撑
- 默认采用**深度分析风格**：结构化章节、系统级拆解、精准术语
- **先下结论，再举实例**：每个章节以论点句开头，再用具体案例和数据展开论证。不要先铺垫再收束
- 正面评价要具体到机制层面，批评要明确指出设计缺陷而非笼统否定
- 格式惯例见 REFERENCE.md 风格指南

### 持续维护项目的特殊规则

如果评测对象是 `status: "ongoing"` 或 `status: "maintained"` 的持续运营游戏（如赛季制网游、长期更新游戏）：

- **保留更新内容概览**：每次赛季/版本更新必须有「更新内容概览」表格，列出新增地图、武器、模式、系统优化的具体条目
- **保留版本对比**：用表格并排对比不同版本/赛季的关键维度变化（武器平衡、产能、局外养成等）
- **不要过度总结**：不要用一两句话概括一次大版本更新——保留原文的逐项拆解和具体数据
- **更新日期**：每次更新评测必须更新 `updatedAt` 字段为当前日期

### 5. 修改确认
在对话中展示完整评测草案后，**必须主动询问用户**：

> 评测文章是否需要修改？

根据用户反馈调整正文、评分或 frontmatter。可以多轮迭代直到用户满意。**用户明确表示"没问题"或"可以"之后，才能进入下一步。**

### 6. 配图确认
修改确认通过后，**必须主动询问用户**：

> 是否有图片需要添加到评测中？

- 如果用户没有图片：`images` 设为 `[]`，直接跳至第 7 步
- 如果用户提供图片：
  1. 将图片放入 `public/images/game-reviews/{slug}/`
  2. 检查图片大小——如果单张超过 **500KB** 或全部合计超过 **2MB**，**主动询问用户**：

  > 图片较大（列出具体数值），是否需要压缩？压缩可大幅减小文件体积，对画质影响轻微。

  3. 用户确认后执行压缩：用 `npx sharp-cli` 转为 WebP（`--quality 85`），删除原文件，更新 `images` 路径为 `.webp`
  4. 图片就位后进入第 7 步

### 7. 写入文件并推送
配图确认后，写入 `content/game-reviews/{slug}.mdx`：
- slug：英文游戏名 kebab-case（如 `cyberpunk-2077`），中文游戏由用户指定
- images：根据第 6 步的结果填写——有图片则列出路径，无图片则 `[]`
- 新建评测：`draft: false`
- 更新评测：保留原 `publishedAt`，更新 `updatedAt` 为当前日期

写入后立即 `git add` + `git commit` + `git push`，提交信息格式 `feat: 新增《游戏名》游戏评测`。

## 模型建议

默认使用当前会话模型。如果评测预期正文超过 100 行，建议使用 Opus 以获得更高质量的深度分析。
